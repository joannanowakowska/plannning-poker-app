# Architecture Decisions

## Context

This is a frontend-only planning poker application.

The backend already exists and cannot currently be changed. The frontend must integrate with the existing WebSocket API described in `docs/planning-poker-api-contract.md`.

The product is an MVP. It has no authentication, no Jira integration, no saved voting history, no custom decks, and no room persistence after everyone leaves.

## Decisions

### 1. Use React, NodeJS And Vite For The Frontend

Decision: The frontend will be built with React, Node and Vite.

React is a good fit for the live room UI because the app needs to react to WebSocket state changes, render participant/vote state, and update controls based on host status.

Vite keeps the MVP lightweight and simple to run locally.

NodeJS is a good fit for the frontend because it is widely used, has a large ecosystem, and is easy to run locally.

We also want to use SCSS for styling, which is supported by Vite rather than a library like tailwind.

Consequences:
- The app can be developed locally with a fast dev server.
- The app can later be deployed as static assets.
- No server-side rendering is required for the MVP.

### 2. Run Locally For Now

Decision: The application will run on localhost during the current stage of the project.

Production hosting is not decided yet.

Consequences:
- The team can focus on product and frontend implementation first.
- Deployment-specific concerns can be deferred.
- Any production URL, routing, and hosting rewrites will need to be decided later.

### 3. Use The Existing Backend As Fixed Infrastructure

Decision: The frontend will connect directly to the existing WebSocket backend.

The frontend will not introduce another backend, proxy, database, or persistence layer for MVP.

Consequences:
- The frontend must follow the existing backend API contract.
- Backend room state, host assignment, voting, reveal, reset, and disconnect behavior are authoritative.
- Product behavior that the backend cannot support must either be handled in the frontend or deferred.

### 4. Use Backend State Snapshots As The Source Of Truth

Decision: The frontend will treat backend `state` messages as the authoritative room model.

Every `state` message is a full snapshot. The frontend will replace room state with each new snapshot.

The frontend may keep local UI-only state, such as:
- Current connection status
- Display name input
- Form validation errors
- Error messages
- Whether the user is on the join screen or room screen

Consequences:
- Users see room state that matches the backend.
- Vote values remain hidden until the backend marks the room as revealed.
- UI components should not independently maintain persistent room state.

### 5. Generate Room IDs In The Frontend

Decision: Room IDs will be generated by the frontend.

The backend creates rooms implicitly when the first user connects with a valid room ID. There is no separate create-room API.
